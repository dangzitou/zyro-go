# AI Agent 面试 Q&A

## 1. 一句话介绍项目

推荐回答：

> 我把黑马点评从一个偏 Redis 高并发的本地生活后端，升级成了一个面向真实业务场景的企业级 Agent 后端。它不是单纯接一个大模型聊天，而是把「语义规划、工具调用、会话记忆、静态 RAG、位置感知推荐、流式输出、限流审计」组合成一套可控系统，核心原则是动态事实走工具，静态规则走 RAG，模型只负责理解、编排和表达。

## 2. 这个项目最重要的设计边界是什么

推荐回答：

> 最重要的是事实边界。门店、价格、营业时间、评分、优惠券、库存这些动态事实，必须走工具和数据库，不能让模型猜，也不能直接塞进 RAG。RAG 只负责静态规则、业务说明和系统约束。这样系统回答才可解释、可审计，也更像企业项目，而不是会说话的 demo。

## 3. 你为什么不把实时门店信息直接放进 RAG

推荐回答：

> 因为实时门店信息变化太快，RAG 很容易过期。比如价格、距离、券、营业状态这些一旦过旧，用户看到的推荐就是错的。企业系统里最怕这种“看起来像懂了，其实答错了”的情况，所以我明确把动态事实和静态知识拆开，动态信息走工具查询，RAG 只做规则增强。

## 4. 你这个 Agent 是怎么分层的

推荐回答：

> 我把它拆成四层：
> 1. Planner 层：先把自然语言问题变成结构化执行计划。
> 2. Tool / DB 层：查询门店、优惠券、博客、地理位置等真实业务事实。
> 3. RAG 层：只注入静态 guide 和约束知识。
> 4. LLM 层：负责理解意图、组织结果、生成最终答案。
>
> 这样每层职责都很清楚，出了问题也能快速定位是规划错了、工具没命中，还是知识边界没划清。

## 5. 你做这个项目时，真正像企业项目的地方在哪里

推荐回答：

> 我觉得不是“接了大模型”就算企业级，关键是有没有真实约束。我这里重点补了几类约束：
> - 动态事实全部工具化，不允许模型自由发挥
> - 位置推荐必须绑定真实坐标，不能假装“附近”
> - 有会话记忆、限流、审计、trace 和工具链追踪
> - 出现无命中时，优先保守回答，不返回看起来像对其实错的结果
> - 做了联调、压测、极端提示词回归，不停留在 happy path 演示层面

## 6. 这轮升级里最有面试价值的一批问题是什么

这是这次最值得讲的一段，因为它能体现你不仅会“做功能”，还会“排障、定边界、做工程化”。

### 6.1 LLM 看起来没调用，实际问题是什么

问题现象：

> 我一开始看 provider 控制台，发现几乎没有调用记录，以为是大模型根本没接进来。

排查过程：

> 后来我把链路拆开看，发现有两层原因。第一层是推荐类问题大部分被后端预取逻辑短路了，也就是工具层已经查到结果，直接返回业务答案，根本不会走主模型回答。第二层是确实有一部分该走模型的链路报错了。

最终结论：

> 真正的问题不是“完全没走 LLM”，而是“推荐类请求被后端预取提前结束了；真正走模型的地方又有协议兼容问题”。这类问题企业里很常见，不能只看控制台调用数，要沿着完整链路看 planner、tool、chat 三层各自有没有真正发请求。

### 6.2 为什么会出现 404 / LLM 不可用

问题现象：

> planner 和主 chat 在一开始持续报 404。

根因：

> 不是模型能力不行，而是 OpenAI 兼容网关的 base-url 配置不对，项目原来还在走旧 provider，而且路径级别配置和 Spring AI 的自动拼接策略不一致。

解决办法：

> 我先单独用 HTTP 请求探测 `/v1/models` 和 `/v1/chat/completions`，确认 provider 的真实接口形态，再把 `spring.ai.openai.base-url` 改成根路径，让 Spring AI 自己拼 `/v1/...`，同时替换成正确可用的 key 和模型名。

面试表达：

> 这个问题本质上不是 AI 问题，而是网关接入问题。我的处理方式不是盲猜，而是先用最小请求验证 provider 真实行为，再反向修正 Spring AI 配置。

### 6.3 为什么又出现了 `text/event-stream` 解码失败

问题现象：

> base-url 改对之后，404 消失了，但同步 planner 和同步 chat 还是失败，报错是 `no suitable HttpMessageConverter found for content type [text/event-stream]`。

根因：

> 这个 provider 的非流式接口也返回了 `text/event-stream`，而 Spring AI 默认的 `call().content()` / `call().entity()` 路径期待的是普通 JSON，因此解码失败。

解决办法：

> 我没有为某一个 provider 单写特殊适配，而是继续沿用 Spring AI 的统一能力，把同步 planner 和同步 chat 全部改成内部走 `stream().content()` 聚合文本。这样仍然是同一套 Spring AI，只是把同步调用改成流式收齐，兼容了这类 provider。

为什么这个解法更好：

> 这样保持了 provider 热插拔能力，没有引入 provider 专属分支；后面换别的 OpenAI-compatible 网关也能继续复用。

### 6.4 为什么推荐类请求有时候会“看起来没走模型”

问题现象：

> 明明已经接了 LLM，但某些推荐问题还是像后端模板化返回。

根因：

> 因为推荐类场景里我做了预取层，先查真实业务数据。如果工具层已经足够回答，系统会优先用业务事实兜底，避免模型在动态事实层自由发挥。

解决办法：

> 后来我把这条链路改成：推荐类问题先做 prefetch，拿到真实门店结果后，再把这些事实注入到 system/prefetch context 里，让模型组织最终回答。这样既有事实可靠性，又有模型表达能力。

面试表达：

> 这体现的是“模型增强业务”，不是“业务依赖模型胡猜”。企业系统里，事实可信度比语言花哨更重要。

## 7. 语义理解这轮具体优化了什么

### 7.1 从“删词拼词”改成“结构化计划驱动”

之前的问题：

> 一开始很多 query 是靠自然语言删词得到的，容易出现脏 query，比如“的30以内的快餐”“找个 餐厅”这种字符串，导致召回很不稳定。

现在的做法：

> 先通过 planner 产出结构化计划 `AgentExecutionPlan`，里面包含：
> - intent
> - city
> - locationHint
> - nearby
> - category / subcategory
> - pricePreference
> - qualityPreference
> - partySize
> - budgetMax
> - excludedCategories
> - negativePreferences
>
> 然后推荐层不再直接删词，而是按这些结构化字段组装查询和过滤条件。

### 7.2 地点解析为什么要改成“保守模式”

之前的问题：

> 地点解析太激进，像“给我找个附近吃饭的地方”这种口语，容易把“找个”错误识别成 locationHint。

现在的做法：

> 我把 `LocationTextParser` 改成保守提取：
> - 优先识别明确的“附近/周边/一带”结构
> - 再识别城市
> - 最后截取地点短语
> - 对明显噪声词做黑名单兜底

效果：

> 现在“找个”“为什么”“这个项目”这类词不会再被当成地标，减少了很多脏 plan。

### 7.3 否定偏好为什么值得讲

问题现象：

> 例如“广州正佳附近不要火锅，想吃点清淡的，两个人，80以内”，之前会误把“火锅”理解成正向类别，完全反了。

优化方式：

> 我给 plan 增加了：
> - `excludedCategories`
> - `negativePreferences`
>
> 这样“不要火锅”会被归到排除项里，“清淡”会进入正向偏好，推荐层再按排除类别过滤，而不是把“火锅”塞进关键词。

面试表达：

> 这个点很像企业真实用户表达，不会每个人都用很标准的搜索语言。支持否定约束是 Agent 从 demo 走向实用化的重要一步。

## 8. 推荐层你是怎么加强的

推荐层现在除了原来的预算、距离、评分，还增加了这些能力：

- 子类目匹配：快餐、咖啡、奶茶、家常菜等
- 排除类别过滤：如排除火锅
- 负向偏好过滤：如太贵、太辣
- 偏好加权：如出餐快、清淡、适合约会、性价比高

推荐回答：

> 我没有把所有偏好都堆给模型，而是把可结构化的约束尽量收回到业务层。比如预算、距离、子类目、排除类别这些由推荐服务显式过滤，像“适合约会”“清淡”“出餐快”这类偏好再作为加权因子参与排序。这样既保留了推荐的业务可控性，也降低了模型误判风险。

## 9. 这轮怎么做回归验证

推荐回答：

> 我没有只测 happy path，而是固化了一批正常+极端提示词。比如：
> - 广州五山附近的人均30以内的快餐店推荐
> - 体育西? 便宜点 吃饭
> - 广州正佳附近不要火锅，想吃点清淡的，两个人，80以内
> - 广州正佳!!!预算80，两个人，约会，安静一点，别太贵
> - SM附近来点学生党喝得起的咖啡或者奶茶
> - 给我找个附近吃饭的地方
>
> 我重点验证四件事：
> 1. 不能再出现 `Agent 调用失败`
> 2. 推荐类短句不能再误路由
> 3. 地点和关键词不能再出现明显脏值
> 4. 否定偏好和预算要能落到结构化 plan 里

## 10. 这轮改完之后，效果提升体现在哪

推荐回答：

> 这轮之后最明显的提升不是“答得更花”，而是“错得更少”。以前复杂口语、极端提示词、否定偏好很容易把链路打崩或者把意图理解反。现在：
> - planner 和主 chat 都已经打通
> - 推荐类问题不会再轻易掉到 `general/factual_lookup`
> - 否定偏好、预算、人数、细类目能进入结构化 plan
> - 极端短句和口语噪声不再轻易把地点解析搞脏

## 11. 这轮之后还有什么残余问题

推荐回答：

> 现在更多已经不是链路问题，而是数据供给问题。比如：
> - 某些城市/商圈下高质量门店样本还不够厚
> - 某些细类目比如“清淡简餐”“学生党奶茶”仍然受限于数据覆盖
> - 缺城市时默认继承当前用户城市，这是一种产品策略，不一定对所有场景都最优
>
> 换句话说，系统已经更像企业系统：先把理解和链路做稳，再把数据厚度一点点补上。

## 12. 如果面试官问“这轮你最满意的地方”

推荐回答：

> 我最满意的不是接通了大模型，而是把一批非常真实的工程问题收住了。比如 provider 的协议兼容、Spring AI 的流式收口、推荐类预取和模型表达的边界、语义理解从删词拼词走向结构化计划、以及极端提示词回归。这些都比“接了个 AI 接口”更能说明我在做一个能落地的系统。

## 13. 一句话总结这轮升级

推荐回答：

> 这轮我不是在继续堆 AI 名词，而是把 Agent 的语义理解、事实边界和模型调用链真正打磨成了一个更可控、更稳定、也更适合企业业务落地的系统。
