# AI Agent 面试 Q&A

## 1. 一句话介绍项目

推荐回答：
> 我把黑马点评从一个偏 Redis 高并发的本地生活后端，升级成了一个面向真实业务场景的企业级 Agent 后端。它不是简单接一个大模型聊天接口，而是把语义规划、工具调用、会话记忆、静态 RAG、位置感知推荐、流式输出、限流审计等能力组合成一套可控系统，核心原则是动态事实走工具，静态知识走 RAG，模型只负责理解、编排和表达。

## 2. 这个项目最重要的设计边界是什么

推荐回答：
> 最重要的是事实边界。门店、价格、营业时间、评价、优惠券、库存这些动态事实，必须走工具和数据库，不能让模型猜，也不能直接塞进 RAG。RAG 只负责静态规则、业务说明和系统约束。这样系统回答才可解释、可审计，也更像企业项目，而不是会说话的 demo。

## 3. 为什么不把实时门店信息直接放进 RAG

推荐回答：
> 因为实时门店信息变化太快，RAG 很容易过期。比如价格、距离、券、营业状态这些一旦过时，用户看到的推荐就是错的。企业系统里最怕这种“看起来像懂了，其实答错了”的情况，所以我明确把动态事实和静态知识拆开，动态信息走工具查询，RAG 只做规则增强。

## 4. Agent 是怎么分层的

推荐回答：
> 我把它拆成四层。第一层是 Planner，把自然语言变成结构化执行计划；第二层是 Tool / DB 层，查询门店、优惠、博客、地理位置等真实业务事实；第三层是 RAG 层，只注入静态 guide 和约束知识；第四层是 LLM 层，负责理解意图、组织结构、生成最终答案。这样每层职责都清楚，出了问题也能快速定位是规划错了、工具没命中，还是知识边界没划清。

## 5. 这个项目像企业级项目的地方体现在哪

推荐回答：
> 不是“接了大模型”就叫企业级，关键是有没有真实约束。我这里重点补了几类约束：动态事实全部工具化，不允许模型自由发挥；位置推荐必须绑定真实坐标，不能假装“附近”；有会话记忆、限流、审计、trace 和工具链追踪；出现无命中时优先保守回答，不返回看起来像对其实错的结果；另外还做了联调、回归和极端提示词轰炸，不停留在 happy path 演示层面。

## 6. 这轮升级里，最有面试价值的一批问题是什么

这部分最值得重点讲，因为它体现的不只是“会做功能”，而是“会定边界、会排障、会工程化”。

### 6.1 为什么一开始看起来像没调起 LLM

推荐回答：
> 一开始 provider 控制台几乎没有调用记录，很像大模型根本没接进来。后面把链路拆开看，发现有两层原因。第一，推荐类问题大部分被后端预取逻辑短路了，也就是工具层已经查到结果，直接返回业务答案，根本不会走主模型。第二，真正需要走 planner 或主 chat 的地方，又叠加了网关兼容问题。所以最后结论不是“完全没走 LLM”，而是“推荐类请求被预取提前结束了，真正走模型的地方又有协议兼容问题”。

### 6.2 为什么会出现 404 / LLM 不可用

推荐回答：
> 一开始 planner 和主 chat 会持续报 404。根因不是模型能力不行，而是 OpenAI-compatible 网关的 base-url 配置不对，项目还在走旧 provider，而且路径级别配置和 Spring AI 的自动拼接策略不一致。我先用最小 HTTP 请求验证 `/v1/models` 和 `/v1/chat/completions` 的真实行为，再反向修正 Spring AI 配置，把 base-url 改成 provider 根路径，让框架自己拼 `/v1/...`，同时换成正确可用的 key 和模型名。

### 6.3 为什么又出现 `text/event-stream` 解码失败

推荐回答：
> 404 解决之后，同步 planner 和同步 chat 还是失败，报错是 `no suitable HttpMessageConverter found for content type [text/event-stream]`。这说明 provider 的非流式接口也返回了 SSE 形式，而 Spring AI 默认的同步 `call().content()` / `call().entity()` 路径期待普通 JSON。我最后没有为某一个 provider 单写特殊适配，而是继续沿用 Spring AI 的统一能力，把同步 planner 和同步 chat 全部改成内部走 `stream().content()` 聚合文本。这样仍然是同一套 Spring AI，只是换了消费方式，后续 provider 也更容易热插拔。

### 6.4 为什么推荐类请求有时看起来没走模型

推荐回答：
> 因为推荐类场景里我做了预取层，先查真实业务数据。如果工具层已经足够回答，系统会优先用业务事实兜底，避免模型在动态事实层自由发挥。后来我把这条链路改成：推荐类问题先做 prefetch，拿到真实门店结果后，再把这些事实注入到 system / prefetch context 里，让模型组织最终回答。这样既有事实可靠性，又保留了模型表达能力。

## 7. 语义理解这一轮具体优化了什么

### 7.1 从“删词拼词”改成“结构化计划驱动”

推荐回答：
> 一开始很多 query 是靠自然语言删词得到的，容易出现脏 query，比如“的30以内的快餐”“找个 餐厅”，导致召回很不稳定。后来我改成先通过 planner 产出结构化计划 `AgentExecutionPlan`，里面包含 intent、city、locationHint、nearby、category、subcategory、pricePreference、qualityPreference、partySize、budgetMax、excludedCategories、negativePreferences。然后推荐层不再直接删词，而是按这些结构化字段组装查询和过滤条件。

### 7.2 为什么地点解析要改成保守模式

推荐回答：
> 地点解析以前太激进，像“给我找个附近吃饭的地方”这种口语，容易把“找个”错误识别成 locationHint。后来我把 `LocationTextParser` 改成保守提取：优先识别明确的“附近 / 周边 / 一带”结构，再识别城市，最后截取地点短语，并对明显噪声词做 blacklist 兜底。这样“找个”“为什么”“这个项目”这类词就不会再被当成地点了。

### 7.3 为什么否定偏好值得专门做

推荐回答：
> 比如“广州正佳附近不要火锅，想吃点清淡的，两个人，80以内”，之前会误把“火锅”理解成正向类别。后来我给 plan 增加了 `excludedCategories` 和 `negativePreferences`。这样“不要火锅”会进入排除项，“清淡”会进入正向偏好，推荐层再按排除类别过滤，而不是把“火锅”塞进关键词。

## 8. 推荐层是怎么加强的

推荐回答：
> 我没有把所有偏好都堆给模型，而是把可结构化的约束尽量收回到业务层。比如预算、距离、子类目、排除类别这些由推荐服务显式过滤；像“适合约会”“清淡”“出餐快”“性价比高”这些偏好，再作为加权因子参与排序。这样既保留了推荐的业务可控性，也降低了模型误判风险。

## 9. 技术取舍说明

这是面试里很容易拉开差距的内容，建议主动讲。

### 9.1 为什么继续选 Spring AI，而不是再切 LangChain4j

推荐回答：
> 我没有为了多接一个框架切到 LangChain4j，而是继续沿用 Spring AI。原因是项目已经跑在 Spring Boot 体系里，memory、advisor、tool calling、OpenAI-compatible provider 都已经打通，继续在这条链上优化一致性最好。企业项目里比起“多接一个框架”，更重要的是技术栈稳定、行为可预期、后续维护成本低。

### 9.2 为什么不为单一 provider 写专属适配层

推荐回答：
> 这是一个很明确的技术取舍。我没有为某一个 provider 单写一个专属网关服务，因为那会破坏大模型调用的热插拔能力，后面接别的 OpenAI-compatible provider 又得再写一套。我最后选择继续用 Spring AI 统一抽象，只是在同步路径里改成内部走 `stream()` 聚合，这样问题是从“消费方式”层面解决的，不是从“绑定某个供应商”层面解决的。

### 9.3 为什么推荐类问题默认弱化知识检索

推荐回答：
> 推荐类问题本质上是位置、预算、品类、优惠等动态约束驱动的问题，核心答案应该来自工具和数据库，而不是知识片段。知识层更适合解释规则、边界和原因，所以我默认弱化 recommendation 的 `useKnowledge`，避免 RAG 把静态内容混进动态推荐主链，降低误导风险。

### 9.4 为什么缺城市时默认继承当前用户城市

推荐回答：
> 这是产品策略上的取舍。像“正佳附近吃什么”“SM附近咖啡店”这种表达，在本地生活场景里大概率就是问自己当前所在城市附近。如果直接拒答，体验会很差。所以我默认继承当前登录用户城市或定位上下文，提升可用性。代价是异地查询可能不完全准确，所以需要在答案和后续优化里把这个边界讲清楚。

### 9.5 为什么无命中时优先保守回答

推荐回答：
> 企业系统里最怕“没数据时硬编结果”。所以我宁可告诉用户当前没有可靠候选，也不乱推一个外地店或者不存在的门店。这个取舍的本质是：可信度优先于表面上的回答完整度。

## 10. 测试与验证过程

这是面试时证明“你不是只做了功能”的关键部分。

### 10.1 我做了哪几层验证

推荐回答：
> 我做了三层验证。第一层是静态和单元测试，验证 planner、地点解析、推荐过滤规则；第二层是真实接口联调，跑 `POST /ai/agent/chat` 和 `POST /ai/agent/chat/stream`；第三层是极端提示词轰炸，专门测口语、省略、噪声、否定偏好和边界场景。

### 10.2 我重点验证哪些点

推荐回答：
> 这轮我重点验证四件事：  
> 1. 不再出现 `Agent 调用失败`  
> 2. 推荐类短句不再误路由到 `general` 或 `factual_lookup`  
> 3. `locationHint` 和 `keyword` 不再出现明显脏值  
> 4. 否定偏好、预算、人数、细分类目要能落到结构化 plan 里

### 10.3 我固化了哪些极端提示词

推荐回答：
> 我专门拿一批真实又极端的句子做回归，比如：  
> - 广州五山附近的人均30以内的快餐店推荐  
> - 体育西？便宜点 吃饭  
> - 广州正佳附近不要火锅，想吃点清淡的，两个人，80以内  
> - 广州正佳!!!预算80，两个人，约会，安静一点，别太贵  
> - SM附近来点学生党喝得起的咖啡或者奶茶  
> - 给我找个附近吃饭的地方

### 10.4 结果怎么说才更有说服力

推荐回答：
> 我不会只说“测过了”，而是会说：这轮改完之后，编译和针对性单测都通过；服务起起来后，再用一组正常和极端提示词做真实接口轰炸，结果里不再出现 `Agent 调用失败`，推荐类问题也能稳定落到 `recommendation + recommend_shops`，同时 planner 返回的结构化字段比之前干净很多。

## 11. 这轮改完之后，效果提升体现在哪

推荐回答：
> 这轮之后最明显的提升不是“答得更花”，而是“错得更少”。以前复杂口语、极端提示词、否定偏好很容易把链路打崩或者把意图理解反。现在 planner 和主 chat 已经打通，推荐类问题不再轻易掉到 `general` 或 `factual_lookup`，否定偏好、预算、人数、细类目也能进入结构化 plan，极端短句和口语噪声不再轻易把地点解析搞脏。

## 12. 这轮之后还有什么残余问题

推荐回答：
> 现在更多已经不是链路问题，而是数据供给问题。比如某些城市和商圈下高质量门店样本还不够厚，某些细类目比如“清淡简餐”“学生党奶茶”仍然受限于数据覆盖。另外，缺城市时默认继承当前用户城市，是一种产品策略，不一定对所有场景都最优。也就是说，系统已经更像企业系统了：先把理解和链路做稳，再把数据厚度一点点补上。

## 13. 如果面试官问“这轮你最满意的地方”

推荐回答：
> 我最满意的不是接通了大模型，而是把一批非常真实的工程问题收住了。比如 provider 协议兼容、Spring AI 的流式收口、推荐类预取和模型表达的边界、语义理解从删词拼词走向结构化计划，以及极端提示词回归。这些都比“接了个 AI 接口”更能说明我在做一个能落地的系统。

## 14. 一句话总结这轮升级

推荐回答：
> 这轮我不是在继续堆 AI 名词，而是把 Agent 的语义理解、事实边界和模型调用链真正打磨成了一个更可控、更稳定、也更适合企业业务落地的系统。
