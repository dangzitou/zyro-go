# AI Agent 面试 Q&A

## 1. 一句话介绍

推荐回答：

> 我把黑马点评这个经典 Redis 高并发项目升级成了一个面向本地生活场景的单 Agent 后端。它不是简单聊天，而是“结构化规划 + 受控工具调用 + 会话记忆 + 静态 RAG + 流式输出 + 限流审计”的组合，核心原则是动态业务事实走工具，静态规则走 RAG，模型负责理解和表达。

## 2. 这个项目最关键的设计边界是什么

推荐回答：

> 最关键的是事实边界。店铺、价格、评分、营业时间、优惠券这类动态业务事实必须走工具和数据库，不能走 RAG。后来我又把探店博客也从 RAG 里拿掉了，因为博客属于半动态内容，也不适合承担稳定知识真相。现在 RAG 只保留静态 guide。

## 3. 这版 RAG 现在收录了什么

推荐回答：

> 当前 RAG 只收录静态规则和解释性文档，也就是 guide，不再收录博客、店铺、优惠券或其他动态内容。

## 4. 为什么连博客也不放进 RAG

推荐回答：

> 因为博客虽然不像价格和库存那样强动态，但它依然有时效性和主观性。如果把博客混进 `retrievalHits`，前后端很容易误把它当成当前事实来源。企业场景里，这种边界不清晰会让排障、治理和业务责任都变得很麻烦。

## 5. 那博客怎么用

推荐回答：

> 博客不再走 RAG，而是走工具或内容接口查询。也就是说，博客属于业务内容能力，不属于静态知识能力。

## 6. 现在整个系统怎么分工

推荐回答：

> 我把系统拆成三层职责：第一层是 Tools/DB，负责动态业务真相；第二层是 RAG，只负责静态规则背景；第三层是 LLM，负责理解问题、编排工具和组织回答。这样职责是清楚的，不会混。

## 7. 为什么这样更适合企业落地

推荐回答：

> 因为企业系统最怕的是事实边界模糊。只要 `retrievalHits` 里混入半动态或动态内容，业务方就会很难判断答案到底是来自规则、博客，还是来自数据库。现在我把 RAG 收敛成 guide-only，系统的可解释性和可治理性会更强。

## 8. 现在 `retrievalHits` 里应该出现什么

推荐回答：

> 现在 `retrievalHits` 里理论上只应该出现 `guide`，不应该再出现 `blog`、`shop` 或 `voucher`。

## 9. 如果面试官问“RAG 还有意义吗”

推荐回答：

> 仍然有意义，只是定位更准确了。它不再承担内容召回或动态事实召回，而是承担静态规则增强，比如 Agent 约束、动态事实约束、解释性背景。这种 RAG 更稳，也更符合企业级系统的边界要求。

## 10. 一句话总结这次调整

推荐回答：

> 我把 RAG 从“博客 + 规则”的混合检索，进一步收敛成了“纯静态 guide 检索”，把博客和动态事实统一回收到工具层，这样整个 Agent 的事实边界更清晰，企业落地风险更低。
