# AI Agent 面试准备

这份文档不是概念秀，而是严格对齐当前 `zyro-go` 最终实现的面试稿。

目标：

- 能讲清楚这版项目到底做成了什么
- 能说明为什么这样做，而不是堆技术名词
- 能承认边界，不把项目吹成“全自动超级智能体”

## 1. 一句话定义

推荐说法：

> 这是一个面向本地生活场景的实用型 `tool-based agent` 后端。我在这版里把它做成了“结构化规划 + 向量 RAG + 受控工具调用”的组合：模型先产出执行计划，再按需走本地向量检索和业务工具，最后生成问答和推荐结果。

不要说：

- “这是一个多智能体系统”
- “这是一个完全自主规划 Agent”
- “这是完整的 RAG 平台”

## 2. 项目升级亮点怎么讲

你可以先讲“升级前后对比”。

升级前：

- 黑马点评原始后端
- 重点在 Redis 高并发链路
- 登录、缓存、秒杀、GEO、Stream、分布式锁

升级后：

- 保留原有 Redis 业务链路
- 升级到 Java 21 + Spring Boot 3.5
- 引入 Spring AI 1.1 做规划式工具调用 Agent
- 增加结构化规划、Redis 会话记忆、本地向量 RAG、推荐工具、trace 回传
- 测试改为默认可在干净环境跑通的单测

推荐总结：

> 我不是把原项目推倒重做，而是在保留原有高并发业务亮点的前提下，把它升级成一个可落地的本地生活 Agent 后端。

## 3. 为什么选 Spring AI

推荐回答：

> 我选 Spring AI 的核心原因是它跟 Spring Boot 生态贴合，够新，也够实用。这个项目本来就是 Java + Spring 体系，我更看重它能直接把 ChatClient、工具调用、会话记忆接进现有 service 层，而不是为了追新去堆一个过重的 Agent 框架。

可以补的点：

- 和现有 Spring 项目融合顺滑
- 配置、Bean、测试方式都熟悉
- 更适合做结构化输出、RAG advisor 和受控工具调用
- 对 Java 面试来说也更容易解释工程落地

## 4. 这版 Agent 的架构怎么讲

建议用六层说法：

### 4.1 Controller 层

- `AiAgentController`
- 负责接收对话请求和清理会话请求

### 4.2 Agent 编排层

- `AiAgentServiceImpl`
- 负责会话 ID、知识补充、系统提示词、Spring AI 调用、trace 汇总

### 4.3 工具层

- `LocalLifeAgentTools`
- 暴露 `search_shops`、`get_shop_detail`、`get_shop_coupons`、`get_hot_blogs`、`recommend_shops`

### 4.4 会话记忆层

- `RedisChatMemoryRepository`
- `MessageWindowChatMemory`
- 按 `prefix:userId:conversationId` 维护短期记忆

### 4.5 规划层

- `AgentPlanningServiceImpl`
- 先输出 `AgentExecutionPlan`
- 决定 `intent`、`preferredTools`、`retrievalQuery`

### 4.6 业务支撑层

- `AiKnowledgeServiceImpl`
- `LocalLifeRagService`
- `ShopRecommendationServiceImpl`
- 原有 `ShopServiceImpl` / `VoucherServiceImpl` / `BlogServiceImpl`

## 5. 为什么它能叫 Agent

推荐回答：

> 因为它不是单纯的聊天接口包一层。我这里先有一个规划步骤，模型会先决定这轮该偏推荐、偏事实查询还是偏内容发现，再结合 RAG 和工具调用去完成回答。严格说它是一个受控场景下的 `tool-based agent`，不是高度自主的复杂智能体。

这个回答里有三个关键词：

- 受控
- 工具调用
- 基于结果继续生成

## 6. 工具调用链路怎么讲

标准链路：

1. 用户输入自然语言问题
2. 服务端补齐系统提示词和记忆
3. 规划器先产出结构化执行计划
4. Spring AI 把 RAG、记忆和可用工具一起提供给模型
5. 模型决定是否触发工具调用
6. 工具层走真实业务 service
7. RAG 提供背景知识，工具提供动态事实
8. 模型基于两者生成最终答案
9. user / assistant 消息写回 Redis 记忆

推荐强调：

> 动态事实由工具给，模型负责理解和表达。

## 7. 工具白名单怎么讲

推荐回答：

> 我没有让模型直接访问 controller 或数据库，而是只暴露一组受控工具。模型的能力边界就是这些 `@Tool` 方法，工具再去调用业务 service。同时我还加了一层结构化规划，让模型优先工具选择也更可控。

可继续补：

- 模型不能随意访问内部 API
- 工具输出字段也是裁剪过的
- `toolTrace` 方便排查到底调用了什么

## 8. 会话记忆怎么讲

推荐回答：

> 我这里做的是 Redis 短期记忆，不是长期用户画像。具体做法是用 `RedisChatMemoryRepository` 做持久化，再用 `MessageWindowChatMemory` 控制窗口大小。会话按 `userId + conversationId` 隔离，并设置 TTL，避免历史上下文无限膨胀。

不要说：

- “我做了长期记忆系统”
- “我实现了复杂语义记忆”

## 9. RAG 这版怎么讲

推荐回答：

> 这版已经上了真正的 RAG，不过我做的是本地向量 RAG，而不是外部向量数据库方案。具体做法是用 `SimpleVectorStore` 把店铺、优惠券、博客和少量规则文档构造成向量索引，再通过 `QuestionAnswerAdvisor` 在对话阶段按需检索。

如果被追问“为什么不上向量库”：

> 因为现在最关键的动态事实还是要靠工具查，所以我优先做了本地向量 RAG，而没有直接上外部向量数据库。这样既把语义检索落地了，也没有把系统复杂度一下拉得太高。

## 10. 这版用了哪些更“新”的 Agent 技术

推荐回答：

> 这版我重点用了三类现在比较主流、而且能落地的 Agent 技术：第一是结构化规划，我会先产出 `AgentExecutionPlan`；第二是向量 RAG，用本地向量库做背景知识增强；第三是递归工具调用，用 `ToolCallAdvisor` 保证模型可以在一轮里继续按需调下一个工具。

## 11. 为什么不做多 Agent / MCP / LangGraph

推荐回答：

> 当前问题主要集中在查店、查券、看热门内容、做推荐，这类任务用单 Agent + 多工具已经够了。多 Agent 和外部 MCP server 会明显增加编排复杂度和调试成本，但对现在这个业务收益不够高，所以我刻意做了技术收敛。

这个回答会显得你是有取舍的，而不是“不会所以没做”。

## 12. 推荐链路怎么讲

推荐回答：

> 推荐不是把全量店铺扔给模型，而是后端先做候选召回、价格过滤、券过滤、距离和热度打分，得到结构化推荐结果，再把前几名结果给模型做解释和交互。

一句话总结：

> 后端负责选，模型负责讲。

## 13. 测试怎么讲

推荐回答：

> 旧版测试强依赖本机 MySQL 和 Redis，回归成本很高。这次升级我把测试改成了默认可在干净环境跑通的单测，重点覆盖 Agent 服务、规划、知识召回和推荐链路。这样即使没有基础设施，也能验证 Agent 主链路有没有回归。

如果面试官问“为什么不做全量集成测试”：

> 当前优先级是先保证关键逻辑可回归。后续如果往线上化推进，我会再补 Redis / MySQL / 秒杀链路的集成测试。

## 14. 边界和不足怎么说

推荐直接承认：

1. 这是单 Agent，不是多智能体系统
2. 会话记忆是短期窗口，不是长期画像
3. RAG 当前是本地向量库，不是外部检索基础设施
4. 推荐链路还是规则和业务特征驱动，不是工业级推荐系统
5. 规划 / 工具 trace 目前主要用于接口返回，还没沉淀成完整观测平台

## 15. 90 秒项目介绍

可以直接背：

> 我把黑马点评这个经典 Redis 项目升级成了一个本地生活 Agent 后端。底层先做了 Java 21 和 Spring Boot 3.5 升级，业务上保留了原有的登录、缓存、秒杀、GEO、Redis Stream 和分布式锁链路。然后在 AI 侧我没有去堆很多复杂框架，而是选了 Spring AI 做实用型落地。这版我做了三个关键增强：第一是结构化规划，先产出 `AgentExecutionPlan`；第二是本地向量 RAG，把店铺、优惠券、博客和规则文档建成向量索引；第三是递归工具调用，保证模型可以按需连续调用多个业务工具。核心原则还是动态事实走工具，背景知识走 RAG，推荐排序走后端，模型负责理解和表达。最后我把测试从依赖本机环境的集成测，改成了默认可跑通的单测，保证 Agent 主链路能持续回归。

## 16. 高频追问

### 15.1 你这个是不是就是“聊天接口 + 几个 API”

推荐回答：

> 如果只是把接口结果拼进 prompt，那确实只是聊天增强。但我这里的关键在于：模型能在受控工具边界内自行选择调用哪个工具、再基于工具结果继续生成，而且会话记忆、知识补充、工具 trace 都是完整接进来的，所以它比简单 API 包装更进一步，但我也不会把它吹成复杂自主智能体。

### 15.2 为什么不用 LangChain4j

推荐回答：

> LangChain4j 当然也能做，但这个项目本身是 Spring Boot 技术栈，我更看重 Spring AI 和现有 Bean、配置、测试体系的贴合度。对这个项目来说，它更轻，也更顺手。

### 15.3 这版升级最大的工程价值是什么

推荐回答：

> 我觉得最大的价值不是“加了 AI”，而是把旧项目升级到了一个现在还能继续迭代的技术底座，同时让 Agent 能真正接业务事实，代码、测试和文档也保持一致。

## 17. 企业级追问怎么答

### 17.1 你这版除了规划和 RAG，还有哪些更偏生产的 Agent 能力？

推荐回答：
> 我后面又补了几块真正影响企业落地的能力。第一是流式输出，我加了 `/ai/agent/chat/stream`，前端可以逐段拿结果。第二是限流，把普通问答和流式问答拆开限流，优先按用户做配额控制。第三是审计留痕，我把 Agent 请求、计划、trace、耗时和状态写进 Redis Stream，方便排障和安全审计。第四是可观测性，接了 Actuator 和 Prometheus，再加上 traceId 日志链路，这样它就不只是一个“能答”的 Agent，而是一个能运维、能排查、能治理的服务。

### 17.2 为什么流式输出值得做？

推荐回答：
> 因为企业里很多 Agent 场景不是一次性短答，而是客服辅助、导购解释、运营分析这类中长回答。流式输出能明显改善首字时间，也方便前端做打字机效果、取消请求和超时处理。所以这不是炫技，而是直接影响可用性的能力。

### 17.3 你怎么做限流，为什么不只在网关层做？

推荐回答：
> 网关层限流我当然也支持，但应用层我还是补了一道，因为 Agent 请求的成本远高于普通 CRUD。我的做法是在 MVC 拦截器层先拦住 `/ai/agent/chat` 和 `/ai/agent/chat/stream`，优先按登录用户计数，拿不到用户再退化到 IP。这样在进入模型编排链路前就能把高成本请求挡掉，能减少模型调用和工具链路的浪费。

### 17.4 审计为什么放 Redis Stream？

推荐回答：
> 因为这个项目本来就重度使用 Redis，高并发链路也已经验证过 Redis 的适用性。审计先放 Redis Stream，一是接入成本低，二是天然支持顺序追加和消费扩展，三是适合后面再接监控、告警、离线分析。这个阶段它是一个轻量但实用的选择，比一开始就引入更重的日志平台更符合项目节奏。

### 17.5 现在这版离真正企业级还有哪些边界？

推荐回答：
> 我会诚实地说它已经具备企业级雏形，但还不是所有治理能力都补完了。比如现在有了限流、审计、Prometheus 和 prod profile，但还没有完整熔断降级、灰度发布、索引增量更新、审计异步消费分析、更加系统化的压测数据。这种回答反而更像真正做过工程的人，而不是只会堆概念。
