# AI Agent 面试 Q&A

## 1. 一句话介绍项目

推荐回答：

> 我把黑马点评从一个偏 Redis 高并发秒杀项目，升级成了一个面向本地生活推荐场景的企业级 Agent 后端。它不是单纯聊天，而是“规划 + 工具调用 + 会话记忆 + 静态 RAG + 位置感知推荐 + 流式输出 + 限流审计”的组合系统，核心原则是动态事实走工具，静态规则走 RAG，模型只负责理解、编排和表达。

## 2. 这个项目解决的真实业务问题是什么

推荐回答：

> 它解决的是本地生活场景下的“可信推荐和可信问答”问题。用户会问附近吃什么、有没有优惠、营业到几点、适不适合约会、预算多少能吃。这里面既有理解问题，也有查动态业务事实，还有位置和预算约束。如果只靠大模型直答，很容易编造；如果只做传统搜索，又不够智能。所以我把它做成了一个受控 Agent。

## 3. 你说它不是玩具项目，核心依据是什么

推荐回答：

> 我认为企业级和玩具项目的区别，不在于有没有接大模型，而在于有没有真实约束。这个项目里我做了几层真实约束：第一，动态事实全部工具化，不能让模型编；第二，RAG 只承载静态规则，避免知识边界混乱；第三，引入登录态、限流、审计、会话记忆和流式输出；第四，位置推荐必须绑定用户坐标，不能假装“附近”；第五，我还做了联调和压测，不是停留在功能演示层面。

## 4. 现在整个系统怎么分层

推荐回答：

> 我把它拆成三层。第一层是 Tools/DB，负责店铺、价格、评分、营业时间、优惠券这些动态业务真相；第二层是 RAG，只负责静态规则、系统约束和解释性知识；第三层是 LLM，负责理解用户意图、选择工具、组织最终回答。这样每层职责都清楚，出了问题也更容易排查。

## 5. 这个项目最关键的设计边界是什么

推荐回答：

> 最关键的是事实边界。店铺、价格、评分、营业时间、优惠券这类动态业务事实必须走工具和数据库，不能走 RAG，更不能让模型自己猜。RAG 只保留 guide 级静态知识，比如 Agent 使用约束、动态事实约束、RAG 范围约束。这样企业落地时，答案的来源是可解释的。

## 6. 为什么你把博客也移出 RAG

推荐回答：

> 因为博客属于半动态内容，它虽然不像价格那样高频变化，但依然带主观性和时效性。如果把博客混进 RAG，前端和业务方很容易误以为它也是当前事实来源。企业系统里最怕这种边界模糊，所以我把博客从 RAG 里拿掉，统一回收到工具层。

## 7. 现在这版 RAG 到底收录了什么

推荐回答：

> 当前 RAG 是 guide-only，只收录静态规则和解释性文档，不再收录博客、店铺、优惠券或其他动态内容。也就是说，RAG 的职责不是帮我找店，而是帮我约束 Agent 的行为边界。

## 8. 如果面试官问“那 RAG 还有意义吗”

推荐回答：

> 有意义，而且更准确。以前很多项目把 RAG 当万能知识库，最后动态事实和静态规则混在一起，很难治理。现在我把 RAG 收敛成规则增强层，它仍然有价值，因为它能稳定给模型注入约束和背景，但不会污染业务事实。

## 9. 工具调用这一层你做了什么

推荐回答：

> 我给 Agent 暴露了几类工具：店铺搜索、店铺详情、优惠券查询、热门博客查询、位置感知推荐。规划器会先判断用户问题属于推荐、事实查询还是内容发现，再给出 preferredTools。后端会优先做工具预取，如果已经能用业务数据直接回答，就不强行走模型自由发挥。

## 10. 位置能力是怎么做的

推荐回答：

> 我给用户资料补了完整地址和坐标字段，不再只存 city。现在 Agent 做“附近推荐”时，会自动读取当前登录用户的地址和经纬度，然后把位置带入推荐服务。这样“附近”就不是文案层面的附近，而是真正按距离计算过的附近。

## 11. 为什么位置能力在这个项目里很重要

推荐回答：

> 因为本地生活推荐最容易出现“语义对了、业务错了”的问题。比如用户在广州天河问附近咖啡店，如果系统推荐了北京王府井的店，那从大模型角度看好像也回答了“咖啡”，但从业务角度完全不可用。所以位置约束是企业级本地生活 Agent 的基础能力。

## 12. 你怎么避免“附近推荐”乱推荐外地店

推荐回答：

> 我做了两个动作。第一，用户有位置时，推荐服务会计算真实地理距离；第二，我加了距离收口阈值，50 公里外的候选会直接剔掉。这样就不会再出现广州用户问附近，结果北京店混进来的问题。

## 13. 如果附近没有满足条件的店，你怎么处理

推荐回答：

> 我没有让系统胡乱退回全库搜索，而是改成一个更保守的企业级策略：如果用户明确问“附近”，又确实有位置，但附近没有命中，就直接告诉他当前附近没有满足条件的候选，并建议放宽预算、换品类或去掉优惠限制。这比给错误结果更可靠。

## 14. 预算能力怎么做的

推荐回答：

> 我补了预算解析规则，像“人均150以内”“预算100元”“不超过200”这类问法都能提取成结构化预算条件，再传给推荐服务做筛选。这样用户自然语言里的预算约束就能真正落到业务过滤层。

## 15. 数据质量你做了什么治理

推荐回答：

> 这块我做得比较实。因为本地生活推荐不怕数据少，怕数据脏。我先导入了大量店铺数据，但后来把像“北京市麦当劳”“广州市肯德基”这种伪地址都清掉了，因为它们根本没法支持附近匹配。后面我保留了能到路级别甚至门牌级别的地址，并做了备份表，保证清洗可回滚。

## 16. 现在店铺数据还有什么现实问题

推荐回答：

> 目前高质量数据已经能支撑一部分真实推荐，但并不是所有类目都很厚。比如广州附近咖啡推荐已经比较像业务系统了，但火锅类在当前高质量数据里样本还不够，所以“附近火锅”更容易出现无命中。这其实是一个很真实的企业问题，不是 Agent 本身坏了，而是数据供给还要继续补。

## 17. 你做了哪些稳定性工作

推荐回答：

> 我做了几个比较企业化的稳定性动作。第一，Agent 接口本身有限流，避免一个用户高并发把模型或工具层打爆；第二，工具和模型失败时有兜底；第三，我修了秒杀订单后台线程消费 Redis Stream 时的 `NOGROUP` 自旋问题，启动时会自动初始化消费组，避免后台异常刷爆 CPU 把 Agent 接口拖慢。

## 18. 你说的那个 Redis Stream 问题，具体是什么

推荐回答：

> 之前 `VoucherOrderServiceImpl` 的后台线程会持续消费 `stream.orders`，但如果 Redis 里没有 `g1` 这个消费组，就会不断抛 `NOGROUP`。这个错误本来和 Agent 业务没直接关系，但它会在后台高频打日志、占 CPU，最后连 Agent 接口时延都被拖高。后来我在启动阶段自动补齐消费组，遇到 `NOGROUP` 也做自愈，这就是典型的企业级稳定性治理。

## 19. 可观测性和审计你做了什么

推荐回答：

> 我做了请求级 trace、Agent 审计记录和工具调用链路记录。每次对话都会留下 conversationId、意图规划、retrievalHits、toolTrace、耗时、状态等信息。这样一旦推荐有问题，我可以回溯到底是规划器判断错了、工具没命中，还是知识层给了错误上下文。

## 20. 限流为什么重要

推荐回答：

> 因为 Agent 项目天然比普通 CRUD 成本高。一次请求可能要经过规划、检索、工具调用、模型生成。如果不做限流，很容易被单用户或脚本打爆。我这里做了用户级限流，压测时也能看到，当并发打得过高时，系统会明确返回“请求过于频繁”，而不是把整个服务拖死。

## 21. 你做过联调吗，联调结果怎么样

推荐回答：

> 做过，而且不是只测 happy path。我实际联调了登录、用户位置、Agent 对话、工具调用和推荐返回。结果上，“附近咖啡店”已经能根据广州天河的用户位置返回广州本地候选，并带距离；“附近火锅店”在当前数据不够厚时，会明确告诉用户附近暂无满足条件的店，而不会给外地结果。

## 22. 你做过压测吗，压测说明了什么

推荐回答：

> 做过两组。第一组是高并发压测，结果能看到部分请求被限流拦下，这说明限流是有效的；第二组是在限流阈值内的基线压测，20 请求、并发 5 的情况下，成功率 100%，平均延迟大概 1.7 秒，P95 大概 2.3 秒。这个结果说明当前系统在小规模并发下是可用的，但如果要更高吞吐，还需要继续优化工具层和模型层。

## 23. embedding 现在是怎么做的

推荐回答：

> 系统支持两套 embedding。理想情况是走真实的远端 embedding 模型，比如 OpenAI 兼容接口；但为了保证项目可运行，我加了 `local-hash` 本地兜底模型。这样即使远端 embedding 网关不可用，RAG 也不会完全失效。不过从企业级效果看，真正上线还是应该切回高质量语义 embedding。

## 24. rerank 你怎么看

推荐回答：

> 我认为 rerank 是值得上的，但不是最优先级第一。现在这版 RAG 规模还不大，先把知识边界、真实 embedding、数据质量和推荐闭环做稳更重要。等 guide 型知识库扩到几十上百条后，`embedding + rerank` 的组合会更有价值。

## 25. 为什么你没有用 LangChain4j

推荐回答：

> 因为这个项目已经走的是 Spring AI 路线，工具调用、RAG、memory、advisor 这些能力都能承接，继续沿这条线往下做一致性更好。企业项目里，技术栈稳定比“多接一个框架”更重要，所以我没有为了堆栈再切 LangChain4j。

## 26. 这个项目现在离真正企业上线还差什么

推荐回答：

> 我觉得还差三块。第一是数据层，要继续补高质量本地生活样本，尤其广州餐饮；第二是知识层，要把现在的 guide-only RAG 扩成可维护的规则知识库；第三是工程层，要继续补更多自动化测试、灰度策略和更细的监控指标。也就是说，它现在已经不是玩具，但要到企业正式上线，还要继续把数据、知识和运维体系做厚。

## 27. 如果让你总结这个项目的最大亮点

推荐回答：

> 我觉得最大亮点不是“接了大模型”，而是把大模型放在了正确的位置上。动态事实不让它编，位置推荐不让它瞎猜，RAG 不让它越界，系统出问题还能追踪到工具链和审计记录。这种设计更像企业项目，而不是一个会说话的 demo。

## 28. 一句话总结这次项目升级

推荐回答：

> 我把一个经典高并发项目升级成了一个面向本地生活推荐的企业级 Agent 后端，重点不是堆 AI 名词，而是把事实边界、位置能力、推荐可信度、稳定性治理和压测验证真正做进了系统里。
